STEP 1 · 資料匯入與格式

把資料放進 BBrowserX

認識 BBrowserX 支援的檔案格式與資料型態，理解「計數矩陣」的本質，並學會如何建立 Study、加入多個樣本，甚至直接匯入 NCBI GEO 上已發表的公開資料。

Learn the file formats and data types BBrowserX supports, understand what a count matrix really is, and how to build a Study — with multiple samples, or even published public data pulled straight from NCBI GEO.

開場比喻

先講個比喻：檔案格式就像「不同國家的插頭」

想像你帶著一台筆電飛到世界各地。電力（也就是你的「資料」）本質上都一樣，但每個國家的插頭形狀都不同——台灣是扁腳、英國是三方腳、歐洲是圓腳。你需要的不是換一顆新電池，而是一個能對上插座的轉接頭。單細胞資料的檔案格式 (file format) 正是這個道理：同一份「基因表達計數」，Python 的人習慣裝進 .h5ad、R 的人習慣裝進 .rds、10x 儀器則吐出 .h5 或一疊 MTX 檔。內容是同一套資料，只是「插頭」不一樣。

換個角度，格式也像同一份班級點名表被翻成不同語言：中文版、英文版、日文版，內容（誰、哪個基因、多少數量）完全一致，只是排版與封裝方式不同。BBrowserX 的厲害之處，就是它自帶了各國插座的萬用轉接頭——你只要認得手上的「插頭」是哪一種，把對的檔案交給它，剩下的它自己讀懂。這一章的目標，就是讓你一眼認出手上的插頭，並知道該把哪個檔（或哪個資料夾）遞過去。

Picture flying around the world with your laptop. The electricity — your "data" — is essentially the same everywhere, but every country has a different plug shape: flat pins in Taiwan, three fat pins in the UK, round pins in Europe. You don't need a new battery; you need an adapter that fits the socket. A single-cell file format works exactly like this: the very same "gene-expression counts" get packed into .h5ad by Python folks, into .rds by R folks, while a 10x instrument spits out a .h5 or a stack of MTX files. Same data inside — just a different "plug."

Another way to see it: formats are like the same class roster translated into different languages — a Chinese edition, an English edition, a Japanese edition. The content (who, which gene, how many) is identical; only the layout and packaging differ. BBrowserX's superpower is that it ships with a universal adapter for all those sockets — you just have to recognize which "plug" you're holding, hand it the right file, and it reads the rest on its own. The goal of this chapter is to make you spot your plug at a glance and know exactly which file (or folder) to pass over.

核心概念

一、什麼是計數矩陣 (count matrix)？

把計數矩陣想成一張超大的「細胞 × 基因」點名表：想像一間可容納數十萬人的體育館，橫排坐著一個個細胞、直欄列著兩萬多個基因，每個格子填的是「這個基因在這個細胞裡被抓到幾個分子」。這就是所有 scRNA-seq 分析的起點——計數矩陣 (count matrix)。

更精確地說，它是一個基因 × 細胞的表格（在 BBrowserX 內部，列 row 常是基因、行 column 常是細胞），每一格記錄「某個基因在某個細胞裡被偵測到的 UMI（Unique Molecular Identifier，獨特分子標籤）數量」。UMI 是黏在每一條 mRNA 分子上的小條碼，作用像演唱會入場的手環——同一個人（同一條原始分子）不管被 PCR 複製幾次，都只算一次，避免重複灌水。一個典型的實驗，大概是 2 萬多個基因 × 數千到數百萬個細胞。

這裡有個關鍵事實：絕大多數基因在絕大多數細胞裡根本沒被偵測到（值為 0）。這不是壞事——一個 T 細胞本來就不會表現 B 細胞的基因。這種「格子大多是空的」特性，讓矩陣變成稀疏 (sparse) 的——單細胞資料通常有 90% 到 95% 以上都是 0。因此所有格式（.h5ad、10x HDF5、MTX）都聰明地只儲存非零值（誰、在哪、多少），空格一律省略，檔案才不會爆炸。舉例：一份 5,000 細胞 × 20,000 基因、若真的把一億個格子全存下來會非常笨重，但因為只有約 5% 非零，稀疏儲存後可能只剩幾十 MB。

最後一個重點——你該交什麼給 BBrowserX？答案是原始／過濾後的整數計數 (raw / filtered counts)，也就是「還沒被加工過的生米」，而不是已經標準化 (normalized) 或取過對數 (log) 的「熟飯」。因為 QC、標準化、批次校正等後續烹調步驟，平台會在裡面自己執行（見 Step 2、Step 3）。若你事先亂加工，反而會讓平台的計算失準。

Think of a count matrix as one enormous "cells × genes" roll-call sheet. Picture a stadium seating hundreds of thousands of people: each row is a cell, each of the 20,000-plus columns is a gene, and every cell of the table says "how many molecules of this gene were caught in this cell." This is where every scRNA-seq analysis begins — the count matrix.

More precisely, it's a genes × cells table (inside BBrowserX, rows are usually genes and columns are usually cells), where each entry records "how many UMIs (Unique Molecular Identifiers) of a given gene were detected in a given cell." A UMI is a tiny barcode stuck onto each mRNA molecule — it works like a wristband at a concert entrance: no matter how many times one person (one original molecule) is duplicated by PCR, they're counted only once, so nobody inflates the headcount. A typical experiment is roughly 20,000+ genes × thousands to millions of cells.

Here's a key fact: most genes are simply not detected in most cells (value = 0). That's not a flaw — a T cell isn't supposed to express B-cell genes. This "mostly empty" property makes the matrix sparse — single-cell data is usually 90–95%+ zeros. So every format (.h5ad, 10x HDF5, MTX) cleverly stores only the non-zero values (who, where, how many) and skips the blanks, keeping files manageable. For example: a 5,000-cell × 20,000-gene matrix has 100 million potential entries — storing them all would be absurd, but since only ~5% are non-zero, sparse storage may shrink it to tens of MB.

One last point — what should you hand BBrowserX? The answer is raw / filtered integer counts, the "raw rice" that hasn't been cooked yet, not values that are already normalized or log-transformed ("cooked rice"). QC, normalization, and batch correction are cooking steps the platform runs internally (see Step 2, Step 3). If you pre-process too aggressively, you can actually throw off the platform's calculations.

記住這句話：「原始計數進、分析結果出」。你只要負責把乾淨、未標準化的整數計數交給 BBrowserX，剩下的 log 轉換、批次校正 (Harmony) 等都在平台內處理。 Remember: "raw counts in, results out." Your job is to hand BBrowserX clean, un-normalized integer counts; log transformation, batch correction (Harmony), and the rest happen inside the platform.
🎯
為什麼重要：理解「稀疏」不只是技術細節。它解釋了為什麼單細胞資料能塞進小小的檔案、為什麼降維（PCA/UMAP）能有效壓縮資訊，也為什麼一個「值為 0」的格子通常代表「沒測到」而非「表達量真的是零」——這在後面的標準化與差異表達會一再出現。 Why it matters: "sparsity" isn't just a technical footnote. It explains why single-cell data fits in tiny files, why dimensionality reduction (PCA/UMAP) can compress the information so well, and why a zero usually means "not detected" rather than "truly zero expression" — a distinction that resurfaces again and again in normalization and differential expression.
矩陣的組成

二、raw vs. filtered、barcodes 與 features 到底是什麼？

要真正看懂 10x 的輸出，你需要認識三個關鍵角色。它們就像一張點名表的「三個座標系統」：barcodes 是每一位學生的學號、features 是每一個科目、matrix 是分數本身。缺任何一個，這張表就讀不懂。

To truly read a 10x output, you need three key characters. Think of them as a roll-call sheet's "three coordinate systems": barcodes are each student's ID number, features are each subject, and the matrix is the scores themselves. Miss any one and the sheet becomes unreadable.

🧾

Barcodes（細胞 ID）

每個細胞被一段獨特的 DNA 條碼「標記」，就像每位客人手上的寄物牌號碼。一份 barcodes.tsv 就是一長串細胞 ID（例如 AAACCTGAGAAACCAT-1），一行一個細胞。矩陣的「行」順序，就靠它對應。

Every cell is tagged with a unique DNA barcode, like a coat-check ticket number in each guest's hand. A barcodes.tsv is just a long list of cell IDs (e.g. AAACCTGAGAAACCAT-1), one cell per line. The matrix's "column" order maps to this list.

🧬

Features（基因）

每一列代表一個基因（feature）。features.tsv（舊版叫 genes.tsv）通常有兩欄：Ensembl ID（如 ENSG00000167286）與基因符號（如 CD3D）。CITE-seq 資料還會多一欄標明是「Gene Expression」還是「Antibody Capture」。

Each row is a gene (feature). features.tsv (older name: genes.tsv) usually has two columns: an Ensembl ID (e.g. ENSG00000167286) and a gene symbol (e.g. CD3D). CITE-seq data adds a third column marking "Gene Expression" vs "Antibody Capture."

🔢

Matrix（計數本體）

matrix.mtx 存的是稀疏矩陣的「非零清單」：第幾列（基因）、第幾行（細胞）、值是多少。它必須靠 barcodes 與 features 兩張座標索引才知道每個數字對應到哪個細胞、哪個基因。

matrix.mtx stores the sparse matrix as a "non-zero list": which row (gene), which column (cell), and the value. It relies on the barcodes and features coordinate indexes to know which cell and which gene each number belongs to.

raw 與 filtered 差在哪？

Cell Ranger 會吐出兩個版本的矩陣，名字很像但意義天差地別，這是新手最常搞混的一步：

  • raw（未過濾）：把所有偵測到的 barcode 都算進來——包含幾十萬個「空油滴」（droplet 裡沒有真細胞，只有背景 RNA）。就像一張把「來賓」和「路過的空氣」都寫進去的名單。檔名常見 raw_feature_bc_matrix。
  • filtered（已過濾）：Cell Ranger 已經用演算法（如 EmptyDrops）判斷哪些 barcode 是真正的細胞，只留下它們。這才是你絕大多數情況要匯入的版本。檔名常見 filtered_feature_bc_matrix。

一個直覺的例子：raw 可能有 737,000 個 barcode（10x 的理論上限），但 filtered 只留下 約 5,000 個真細胞。差距那麼大，正是因為絕大多數油滴是空的。

Cell Ranger emits two versions of the matrix with confusingly similar names but wildly different meaning — this trips up beginners more than anything else:

  • raw (unfiltered): includes every barcode detected — including hundreds of thousands of "empty droplets" (droplets with no real cell, just ambient RNA). It's a guest list that also records "the air that walked past." Often named raw_feature_bc_matrix.
  • filtered: Cell Ranger has already used an algorithm (e.g. EmptyDrops) to decide which barcodes are real cells and kept only those. This is the version you'll import in the vast majority of cases. Often named filtered_feature_bc_matrix.

An intuitive example: raw might contain 737,000 barcodes (10x's theoretical ceiling), while filtered keeps only ~5,000 real cells. The gap is enormous precisely because most droplets are empty.

🧠
專家小撇步：初學者幾乎都選 filtered。只有在你想自己用進階方法（如 SoupX / CellBender）去除背景 RNA「湯汁」污染時，才會刻意載入 raw。看到檔名開頭是 filtered_ 就對了。 Pro tip: beginners should almost always pick filtered. You'd only deliberately load raw when you want to remove ambient-RNA "soup" contamination yourself with advanced tools (like SoupX / CellBender). If the filename starts with filtered_, you're good.
支援格式

三、BBrowserX 支援哪些檔案格式？

單細胞資料在不同工具間流通時，會以幾種標準容器儲存。BBrowserX 直接支援下列四大類，涵蓋 Python (Scanpy)、R (Seurat) 與 10x Cell Ranger 的原生輸出。它們最大的分野其實只有一句話：是「一個檔案打包全部」，還是「一疊檔案一起才算數」。

Single-cell data travels between tools in a handful of standard containers. BBrowserX directly supports the four families below, covering the native outputs of Python (Scanpy), R (Seurat), and 10x Cell Ranger. Their biggest distinction boils down to one line: is it "one file packs everything," or "a stack of files that only counts together"?

🐍

.h5ad (AnnData)

Python / Scanpy 生態系的標準格式，副檔名 .h5ad。以 HDF5 為底——把它想成一個整理得井井有條的行李箱：一個檔案同時裝下計數矩陣、細胞與基因的中繼資料（obs 存細胞資訊、var 存基因資訊）、降維座標 (UMAP/PCA)、分群結果等。單檔、可攜、最常見，也最不容易出錯。

The standard for the Python / Scanpy ecosystem, extension .h5ad. Built on HDF5 — think of it as a neatly packed suitcase: one file holds the count matrix, cell & gene metadata (obs for cells, var for genes), embeddings (UMAP/PCA), clustering, and more. One file, portable, the most common — and the least error-prone.

📘

Seurat 物件 (.rds)

R / Seurat 生態系的物件，通常存為 .rds。可想成 R 世界的行李箱，內含計數、分群結果、metadata、甚至多個 assay（如 RNA 與 ADT）。若你的合作者用 R 分析，直接請他 saveRDS() 存下物件給你即可匯入。

The object of the R / Seurat ecosystem, usually saved as .rds. Think of it as R's version of the suitcase — carrying counts, clustering, metadata, even multiple assays (like RNA and ADT). If a collaborator works in R, just have them saveRDS() the object; it imports directly.

📦

10x HDF5 (.h5)

10x Genomics Cell Ranger 的單檔輸出，檔名常為 filtered_feature_bc_matrix.h5（或 raw_..._bc_matrix.h5）。一個 .h5 檔就把稀疏矩陣＋基因＋barcode 全包在裡面——等於把 MTX 那三個檔壓成一個手提包，方便許多。

The single-file output of 10x Genomics Cell Ranger, typically named filtered_feature_bc_matrix.h5 (or raw_..._bc_matrix.h5). One .h5 file wraps the sparse matrix + genes + barcodes together — it's basically the three MTX files zipped into a single handbag, much more convenient.

🗂️

10x MTX bundle（三檔）

經典的 Matrix Market 資料夾，三個檔缺一不可：matrix.mtx（稀疏計數）＋ barcodes.tsv（細胞 ID）＋ features.tsv 或 genes.tsv（基因）。常見副檔名帶 .gz。像一組三件式行李，匯入時要選整個資料夾，分開就讀不了。

The classic Matrix Market folder — three files, none optional: matrix.mtx (sparse counts) + barcodes.tsv (cell IDs) + features.tsv or genes.tsv (genes). Often .gz-compressed. Like a matching three-piece luggage set: point the import at the whole folder — split them up and it won't read.

它們從哪裡來？看一眼 Cell Ranger 的 outs/ 資料夾

10x 的資料經過 Cell Ranger（比對＋計數的官方軟體）處理後，會產生一個 outs/ 資料夾。你要找的兩個「金礦」，都在裡面：

After 10x data passes through Cell Ranger (the official align-and-count software), you get an outs/ folder. The two "gold nuggets" you're looking for both live inside:

sample_A/outs/
├── filtered_feature_bc_matrix.h5 # ← 單檔，最省事 / single file, easiest
├── filtered_feature_bc_matrix/ # ← MTX 三檔資料夾 / the MTX trio
│   ├── matrix.mtx.gz
│   ├── barcodes.tsv.gz
│   └── features.tsv.gz
├── raw_feature_bc_matrix/ # ← 含空油滴，通常不用 / has empty droplets, usually skip
├── web_summary.html # ← QC 報告，值得先看 / QC report, worth a look
└── molecule_info.h5
⚠️
MTX 最容易出錯：三個檔必須在同一個資料夾、名稱正確、且屬於同一個樣本。若你自己重新命名、或把不同樣本的檔混進同一夾，匯入就會失敗。單檔格式（.h5ad / .h5 / .rds）通常最不容易出問題——能拿到 .h5 就別跟自己過不去。 MTX trips people up most: all three files must sit in the same folder, be correctly named, and belong to the same sample. Renaming them yourself, or mixing files from different samples into one folder, breaks the import. Single-file formats (.h5ad / .h5 / .rds) are usually the least error-prone — if you can grab the .h5, don't make life harder.

Study 與資料型態

六、建立 Study、加入多樣本、匯入公開資料

把 Study（研究專案）想成一本相簿，每個樣本 (sample) 是相簿裡的一張照片。你通常會先建立一本相簿（新的 Study），再把一張張照片（樣本）夾進去。在 BBrowserX 裡的操作邏輯就是這樣：先 New Study，再 Add sample。

為什麼要把多個樣本放進同一本相簿？因為只有在同一個 Study 裡，你才能把它們放上同一張 UMAP、直接比較（例如處理組 vs. 對照組、腫瘤 vs. 正常組織）。但這也帶來一個經典難題：不同時間、不同 10x lane、不同病人的樣本，會夾帶各自的技術性色差——就像用不同相機、不同光線拍的照片，色調會不一致。這種色差叫批次效應 (batch effect)，若不處理，它會偽裝成「生物差異」騙過你的分析。這正是為什麼平台會在後續用 Harmony 幫你把各樣本「對齊色調」（見 Step 3）。

還有個好消息：你不一定要自己跑實驗產生資料——BBrowserX 也能直接匯入已發表的公開資料集。最大的寶庫就是 NCBI GEO（Gene Expression Omnibus），全世界的研究者把論文用到的原始資料存在那裡供人重現。下一段就帶你走一遍。

Think of a Study (project) as a photo album, and each sample as a photo in it. You usually create the album first (a new Study), then slip the photos (samples) in one by one. That's the logic in BBrowserX: first New Study, then Add sample.

Why put several samples in the same album? Because only within one Study can you place them on the same UMAP and compare them directly (e.g. treated vs. control, tumor vs. normal). But that raises a classic problem: samples from different days, 10x lanes, or patients carry their own technical color casts — like photos shot on different cameras under different lighting, the tones don't match. That cast is called a batch effect, and left unaddressed it disguises itself as "biological difference" and fools your analysis. This is exactly why the platform later uses Harmony to "color-match" your samples (see Step 3).

Good news too: you don't have to run your own experiment to get data — BBrowserX can import published public datasets directly. The biggest treasure trove is NCBI GEO (Gene Expression Omnibus), where researchers worldwide deposit the raw data behind their papers for others to reproduce. The next section walks you through it.

BBrowserX 能處理的單細胞資料型態

🧫

scRNA / snRNA-seq

單細胞與單細胞核 RNA 定序——最核心的基因表達計數，是整個流程的主軸。

Single-cell and single-nucleus RNA-seq — the core gene-expression counts and the backbone of the whole pipeline.

🛡️

TCR / BCR

🎯

🏷️

🔀

🌐

直接匯入 NCBI GEO 等已發表資料集，重現或延伸他人研究。

Import published datasets from NCBI GEO and beyond, to reproduce or extend others' work.

💡
先想好「Study 的邊界」：把你打算一起比較的樣本放進同一個 Study（例如處理組 vs. 對照組）。分屬不同 Study 的資料無法在同一張 UMAP 上直接比較。規劃時可以自問：「這幾個樣本，我最後想不想放在同一張圖上講故事？」想，就放同一個 Study。 Decide your Study's scope first: put the samples you intend to compare together into one Study (e.g. treated vs. control). Data split across separate Studies can't be compared on the same UMAP. A useful question while planning: "Do I ultimately want to tell one story with these samples on the same figure?" If yes, one Study.
實戰走查

七、實戰：從 NCBI GEO 匯入一份已發表資料

假設你讀到一篇很棒的論文，想親手重跑它的單細胞資料。作者幾乎一定會把資料存在 GEO，並在論文的「Data Availability」段落給你一組編號（像 GSE123456）。跟著下面五步走，你就能把它變成 BBrowserX 裡一個可分析的 Study。

Say you read a great paper and want to re-run its single-cell data yourself. The authors almost always deposit it in GEO and give you an accession number (like GSE123456) in the paper's "Data Availability" section. Follow these five steps to turn it into an analyzable Study in BBrowserX.

找到 GEO 編號 (accession)

在論文末尾或方法段找 GSExxxxxx。到 ncbi.nlm.nih.gov/geo 搜這串編號，打開該研究的頁面。

Look for a GSExxxxxx in the paper's end matter or methods. Search that string at ncbi.nlm.nih.gov/geo and open the study page.

看 Supplementary files 提供什麼

捲到頁面底部的 Supplementary file 區。你希望看到的是能對上前面四大格式的檔案：最理想是 *_filtered_feature_bc_matrix.h5（單檔），其次是一組 matrix.mtx.gz + barcodes.tsv.gz + features.tsv.gz。

Scroll to the Supplementary file section at the bottom. You want files that map to one of the four formats above: ideally a *_filtered_feature_bc_matrix.h5 (single file), otherwise a set of matrix.mtx.gz + barcodes.tsv.gz + features.tsv.gz.

下載並依樣本整理資料夾

下載後，若是 MTX，請每個樣本各自一個資料夾，並把檔名整理成標準的 matrix.mtx.gz / barcodes.tsv.gz / features.tsv.gz——GEO 常會在檔名前加一堆前綴（如 GSM111_S1_matrix.mtx.gz），要把前綴去掉、拆進各自資料夾。

After downloading, if it's MTX, give each sample its own folder and rename files to the standard matrix.mtx.gz / barcodes.tsv.gz / features.tsv.gz — GEO often prepends clutter (like GSM111_S1_matrix.mtx.gz), so strip the prefixes and split them into per-sample folders.

建立 Study 並逐一 Add sample

在 BBrowserX 新建一個 Study，把整理好的每個樣本依序加入（.h5 選檔、MTX 選資料夾）。替每個樣本取有意義的名字（如 Tumor_P1、Normal_P1），日後比較才不會混。

Create a new Study in BBrowserX and add each tidied sample in turn (pick the file for .h5, pick the folder for MTX). Give each sample a meaningful name (like Tumor_P1, Normal_P1) so comparisons stay clear later.

讓平台接手 QC 與整合

資料進來後，就交給後續步驟：QC 過濾（Step 2）、標準化與 Harmony 批次校正（Step 3）。你已完成最關鍵的一步——把對的資料，用對的方式，放進了對的地方。

Once the data is in, hand it to the later steps: QC filtering (Step 2), normalization and Harmony batch correction (Step 3). You've done the most critical part — putting the right data, the right way, in the right place.

🧭
實戰情境：GEO 上偶爾只提供作者已經標準化／log 過的矩陣，或非標準的 .txt / .csv 表格。這時要多留意：優先找原始整數計數；若只有處理過的值，分析結果會受限，且務必在報告中註明資料來源與處理狀態。 Real-world scenario: GEO sometimes only offers an author's already-normalized/log matrix, or a non-standard .txt / .csv table. Be careful here: prefer raw integer counts; if only processed values exist, your analysis is constrained — and always note the data's source and processing state in your report.
除錯

八、常見匯入錯誤與怎麼救

就算格式選對了，還有幾個「隱形陷阱」會讓匯入後的資料看起來怪怪的。先認識它們，遇到時就不會慌。

Even with the right format, a few "invisible traps" can make the imported data look off. Knowing them in advance means you won't panic when they show up.

🆔

用錯基因 ID 型態

同一個基因有兩種寫法：基因符號（CD3D）與 Ensembl ID（ENSG00000167286）。就像同一個人有「小名」和「身分證字號」。若匯入後你搜尋 CD3D 卻找不到基因，很可能資料存的是 Ensembl ID。解法：確認 features 檔用的是哪一欄，必要時對照轉換。

The same gene has two spellings: a gene symbol (CD3D) and an Ensembl ID (ENSG00000167286) — like a person's nickname vs. national ID number. If you search CD3D after import and find nothing, the data probably stores Ensembl IDs. Fix: check which column the features file uses and map between them if needed.

🧬

混到不同物種

人類基因習慣全大寫（CD3D），小鼠則首字母大寫（Cd3d）。若把人類與小鼠樣本混進同一 Study，或用錯物種的 marker 註釋，結果會一團亂。解法：確認每個樣本的物種一致，marker/參考也要對應同一物種。

Human gene symbols are conventionally ALL CAPS (CD3D), while mouse uses Title Case (Cd3d). Mixing human and mouse samples in one Study, or annotating with the wrong species' markers, produces a mess. Fix: confirm every sample is the same species, and match markers/references to that species.

🔄

矩陣被轉置了

「基因 × 細胞」被存成了「細胞 × 基因」——行列顛倒。徵兆很明顯：軟體以為你有兩萬個「細胞」、卻只有幾千個「基因」。就像把班級點名表橫著讀成直的。解法：匯入前確認方向；多數標準格式方向固定，但自製的 .csv 最容易出這種錯。

"genes × cells" got saved as "cells × genes" — rows and columns swapped. The tell is obvious: the software thinks you have 20,000 "cells" but only a few thousand "genes." Like reading a roll-call sheet sideways. Fix: verify orientation before import; standard formats are usually fixed, but hand-made .csv files cause this most often.

🚑
常見誤區：「匯入沒報錯 = 資料一定正確」。錯！匯入成功只代表格式讀得進去，不代表內容合理。養成習慣：匯入後先看三個數字——細胞數、基因數、每個細胞的 UMI/gene 分布。若細胞數與論文差一個數量級、或基因數只有幾百，八成中了上面某個陷阱。 Common misconception: "no error on import = the data must be correct." Wrong! A successful import only means the format parsed — not that the content makes sense. Build the habit: right after import, check three numbers — cell count, gene count, and the per-cell UMI/gene distribution. If the cell count is an order of magnitude off from the paper, or gene count is only a few hundred, you've likely hit one of the traps above.
